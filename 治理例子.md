### 1.1 Customer 主键字段不一致

**典型现象**

同一个业务对象 `Customer / 客户`，在不同系统中用于识别客户的字段名和识别规则不一致。

| 系统  | 对象  | 识别字段或规则 | 字段名 / 技术名 |
| --- | --- | --- | --- |
| CRM xSales | Customer | 建议以客户编号唯一标识；无编号时用客户名称、业务上下文和发生日期组合识别 | `客户编码 / Customer Code` |
| 授信 Credit Online | Customer | 以客户编号唯一识别客户 | `客户编号 / Customer Number` |
| 市场计划 XIBP | Customer | 以客户编码唯一识别每个客户 | `客户编码 / Customer Code` |
| 质量问题系统 | Customer | 优先以客户名称识别；若存在客户编码，以客户编码作为业务唯一标识 | `客户名称 / customerName`、`客户编码`、`qc_customer_code` |

**治理问题**

- `Customer Code`、`Customer Number`、`客户编码`、`客户编号`、`qc_customer_code` 是否应该统一映射为企业级客户编码。
- 如果某些质量问题记录只有 `customerName`，没有客户编码，是否可以直接并入客户主数据。
- 客户编号相同但客户名称不同，或者客户名称相同但客户编号不同，如何处理。

**可落地治理规则**

| 规则类型 | 建议规则 |
| --- | --- |
| 企业标准字段 | 建立标准字段 `customer_code` 或 `customer_id` |
| 字段别名 | `客户编码`、`客户编号`、`Customer Code`、`Customer Number`、`qc_customer_code` 均作为 `customer_code` 的来源别名 |
| 匹配优先级 | `统一社会信用代码` > `客户编码/客户编号` > `客户名称 + 地区/行业/业务上下文` |
| 弱匹配规则 | 只有客户名称、没有客户编码的记录进入“待匹配客户池” |
| 冲突规则 | 编码相同但名称差异较大时进入人工审核；名称相同但编码不同不能自动合并 |

### 1.2 Customer Name 字段泛化和技术字段不一致

**典型现象**

同样表示客户名称，不同系统有不同字段名。有些系统使用明确的 `Customer Name`，有些系统只叫 `Name`，有些系统使用技术字段 `customerName` 或 `qc_customer_name`。

| 系统  | 字段名 | 风险  |
| --- | --- | --- |
| CRM xSales | `客户名称 / Customer Name` | 语义清晰 |
| 授信 Credit Online | `客户名称 / Customer Name` | 语义清晰 |
| 市场计划 XIBP | `名称 / Name` | 字段名过泛，脱离 Customer 上下文后容易误映射 |
| 质量问题系统 | `客户名称 / customerName` | 技术名与业务名可映射 |
| 质量事件表 | `qc_customer_name`、`qcCustomerName` | 质量事件直接记录客户维度 |
| CRM API | `customerName` | API 请求或响应中的客户名称 |

**治理问题**

- `Name` 是否总是客户名称。答案是否定的，只有在 Customer 对象上下文中才是客户名称。
- `customerName`、`qc_customer_name`、`Customer Name` 是否应统一到同一个企业标准字段。
- 如果系统里存在多个 `name` 字段，如何避免自动血缘识别误判。

**可落地治理规则**

| 规则类型 | 建议规则 |
| --- | --- |
| 企业标准字段 | `customer_name` |
| 上下文约束 | `Name` 只有在对象类型为 Customer 时才能映射到 `customer_name` |
| 技术别名 | `Customer Name`、`客户名称`、`customerName`、`qc_customer_name`、`qcCustomerName` 均映射到 `customer_name` |
| 质量检查 | 字段名为 `name` 的列必须有对象上下文或表上下文，否则标记为“泛名字段待确认” |

### 1.3 Customer Status 同名但语义不同

**典型现象**

多个系统都有“客户状态”或 `Status`，但它们表示的业务含义不同。

| 系统  | 字段名 | 实际含义 |
| --- | --- | --- |
| CRM xSales | `客户状态 / Customer Status` | 客户维护或审批状态 |
| 市场计划 XIBP | `状态 / Status` | 客户是否可用于计划方案和需求分析 |
| 授信 Credit Online | 准入、额度、评级、风险记录 | 是否准入、额度多少、风险如何、是否预警 |
| 质量问题系统 | 未在 Customer 最小属性中直接建 `客户状态` | 更关注客户名称、客户类型、地区部、行业 |

**治理问题**

- `Customer Status` 能否全企业统一为一个字段。
- CRM 的“客户维护/审批状态”与 XIBP 的“计划可用状态”是否能直接比较。
- 授信风险状态是否应该影响计划、订单、应收等系统。

**可落地治理规则**

| 标准字段 | 含义  | 来源  |
| --- | --- | --- |
| `customer_master_status` | 客户主数据维护或审批状态 | CRM |
| `planning_eligibility_status` | 客户是否可进入计划方案和需求分析 | XIBP |
| `credit_onboarding_status` | 客户是否进入授信管理范围 | 授信  |
| `customer_risk_status` | 客户风险等级、预警或处置状态 | 授信/风控 |

**治理检查**

- XIBP 客户状态为“暂停合作”，但 CRM 客户状态为“有效”时，不直接判为脏数据，而是标记为“跨域状态差异”。
- 授信风险为高风险，但订单、计划或应收仍正常使用时，进入“风险规则联动检查”。
- 所有 `Status` 字段必须附带语义域，例如 `master`、`planning`、`credit`、`risk`。

### 1.4 Customer Type 字段同名但用途不同

**典型现象**

不同系统都有 `客户类型 / Customer Type`，但业务用途不同。

| 系统  | 字段名 | 用途  |
| --- | --- | --- |
| 授信 Credit Online | `客户类型 / Customer Type` | 区分客户分类，并影响准入、额度或风险口径 |
| 质量问题系统 | `客户类型 / customerType` | 根据客户名称带出的客户分类 |
| 质量事件表 | `qc_customer_type`、`qcCustomerType` | 质量事件记录中的客户类型字段 |

**治理问题**

- 授信客户类型和质量客户类型是否来自同一套枚举。
- 如果枚举不同，是否还应该都叫 `customer_type`。
- 客户类型是否影响授信准入、质量通报范围、客户声音分析等不同业务规则。

**可落地治理规则**

| 标准字段 | 含义  |
| --- | --- |
| `customer_business_type` | 企业级客户基础分类 |
| `credit_customer_type` | 授信准入、额度、风险口径下的客户类型 |
| `quality_customer_type` | 质量问题场景下由客户选择或配置带出的客户类型 |

**治理检查**

- 同一个客户在授信系统和质量系统中的客户类型不同，不直接判为错误。
- 先确认两个字段是否来自同一枚举表或码表。
- 如果同源，建立码值映射；如果不同源，保留不同语义字段。

### 1.5 统一社会信用代码只在授信侧明确出现

**典型现象**

授信系统中明确有 `统一社会信用代码 / Unified Social Credit Code`，用于校验企业主体身份和避免客户重复；CRM、XIBP、质量系统的 Customer 最小属性中未明确出现该字段。

| 系统  | 是否明确出现统一社会信用代码 | 影响  |
| --- | --- | --- |
| 授信 Credit Online | 是   | 可用于企业主体校验和去重 |
| CRM xSales | 未在 Customer 最小属性中出现 | 主要依赖客户编码和客户名称 |
| 市场计划 XIBP | 未在 Customer 最小属性中出现 | 主要依赖客户编码 |
| 质量问题系统 | 未在 Customer 最小属性中出现 | 主要依赖客户名称或客户编码 |

**治理问题**

- 客户名称相似但编码不同，如何判断是否同一企业主体。
- 质量系统中只有客户名称的记录，如何与授信客户对齐。
- 如果客户编码相同但统一社会信用代码不同，是否说明主数据存在严重冲突。

**可落地治理规则**

| 规则类型 | 建议规则 |
| --- | --- |
| 企业标准字段 | `unified_social_credit_code` |
| 高置信合并 | 统一社会信用代码相同，可以作为高置信合并依据 |
| 禁止自动合并 | 客户名称相似但统一社会信用代码不同，禁止自动合并 |
| 高风险冲突 | 客户编码相同但统一社会信用代码不同，进入人工审核 |
| 数据补齐 | CRM、XIBP、质量系统中的企业客户应逐步补齐统一社会信用代码或建立映射 |

### 1.6 客户地区/行业字段存在拼写和口径问题

**典型现象**

质量系统中客户地区部字段出现业务名、API 名和疑似拼写偏差的技术名。

| 业务含义 | 字段或表 | 问题  |
| --- | --- | --- |
| 客户地区部 | `客户地区部 / customerRegion` | 业务对象中的标准引用名 |
| 客户地区部 | `customerEgion` | 疑似 `customerRegion` 的拼写偏差 |
| 客户地区部配置 | `qc_customer_egion` | 表名疑似 region 拼写偏差 |
| 客户行业 | `客户行业 / customerIndustry` | 与 CRM、授信中的行业字段需要对齐 |

**治理问题**

- `customerEgion` 是否是 `customerRegion` 的历史拼写错误。
- `qc_customer_egion` 是否应作为客户地区部维表或配置表。
- 客户地区、地区部、代表处、办事处是否属于同一层级，还是不同组织维度。

**可落地治理规则**

| 标准字段 | 技术别名 |
| --- | --- |
| `customer_region` | `customerRegion`、`customerEgion` |
| `customer_region_dimension` | `qc_customer_egion` |
| `customer_industry` | `customerIndustry`、`Customer Industry`、`Industry` |

**治理检查**

- 新接口禁止继续使用 `customerEgion`，老接口保留兼容映射。
- 元数据平台中将 `customerEgion` 标记为 deprecated alias。
- `qc_customer_egion` 作为客户地区部配置表纳入码表治理。

### 2.1 采购订单审批流程

**典型现象**

采购订单审批不是一个简单的“提交按钮”，而是由单据身份、订单状态、审批路线、提交角色和后续通知共同构成的流程治理点。

| 治理对象 | 具体字段/动作 | 当前问题 |
| --- | --- | --- |
| 采购订单身份 | `采购订单号 / Purchase Order Number` | 是否全流程都以采购订单号作为唯一标识 |
| 流程状态 | `订单状态 / Order Status` | 草稿、待审批、已确认、变更中、已关闭等状态是否统一 |
| 执行动作 | `提交采购订单审批 / Submit Purchase Order Approval` | 什么时候允许提交审批 |
| 执行角色 | `采购履约人员` | 是否只能由采购履约人员提交 |
| 动作参数 | `提交范围 / Submission Scope` | 本次审批覆盖采购订单、订单行还是变更范围 |
| 动作参数 | `审批路线 / Approval Route` | 审批路线、审批角色或责任链是否标准化 |
| 动作参数 | `提交原因 / Submission Reason` | 提交审批的业务原因是否必填 |

**治理问题**

- 采购订单在什么状态下可以提交审批。
- `审批路线` 是自由文本，还是受控的审批链配置。
- `提交范围` 能否为空；如果涉及订单行或变更范围，是否必须明细化。
- PO 例外被接受后，是直接更新采购订单，还是必须走变更审批。

**可落地治理规则**

| 规则类型 | 建议规则 |
| --- | --- |
| 标准字段 | `po_number`、`po_status`、`supplier_id`、`purchasing_org`、`approval_route` |
| 状态机 | `草稿/已维护 -> 待审批 -> 已通过/已驳回`，禁止跳状态 |
| 前置条件 | 采购订单必要信息未完成时，不允许提交审批 |
| 角色权限 | 默认只有 `采购履约人员` 可提交；其他角色需业务确认 |
| 审批范围 | `submission_scope` 必须说明覆盖订单、订单行或变更范围 |
| 后续通知 | 提交后必须通知审批参与者 |

### 2.2 发票匹配与付款前置控制

**典型现象**

发票流程的关键不是“有没有发票”，而是发票是否完成校验、是否匹配付款请求、是否移交后续处理。

| 治理对象 | 具体字段/动作 | 当前问题 |
| --- | --- | --- |
| 发票身份 | `发票号码 / invoiceNumber` + `供应商` | 不能只靠发票号识别发票 |
| 发票金额 | `发票金额 / invoiceAmount` | 金额是否完整、是否与付款请求一致 |
| 匹配状态 | `发票匹配状态 / invoiceMatchingStatus` | 是否已匹配付款请求 |
| 移交状态 | `发票移交状态 / invoiceHandoverStatus` | 是否完成资料校验并移交 |
| 执行动作 | `匹配发票 / matchInvoice` | 什么时候建立发票与付款请求或批次的关系 |
| 执行角色 | `发票管理员或会计` | 谁有权匹配发票 |
| 动作参数 | `匹配对象 / Match Target` | 匹配到付款请求、付款请求行还是批次 |

**治理问题**

- 未匹配的发票是否允许进入付款流程。
- 已移交但未匹配的发票是否需要自动进入待处理池。
- 不同来源的发票是否需要独立业务流程。
- 红字确认单、重复确认、后续处理范围是否需要单独治理。

**可落地治理规则**

| 规则类型 | 建议规则 |
| --- | --- |
| 标准主键 | `invoice_number + supplier_id` |
| 付款闸门 | `invoiceMatchingStatus = 未匹配` 时不允许付款 |
| 移交闸门 | `invoiceHandoverStatus = 未移交` 时只能处于资料校验或补充阶段 |
| 匹配对象 | `match_target` 必须指向付款请求、付款请求行或批次之一 |
| 角色权限 | 只有发票管理员或会计可以执行 `matchInvoice` |
| 来源分流 | 建立 `invoice_source_type`，判断不同来源是否走独立流程 |

### 2.3 工单执行与物料发放控制

**典型现象**

制造现场的工单执行需要把工单状态、当前工序、齐套状态、库存状态和物料发放动作联动起来。

| 治理对象 | 具体字段/动作 | 当前问题 |
| --- | --- | --- |
| 工单身份 | `工单号 / Work Order Number` | 是否只用工单号识别，还是要组合生产订单和工艺版本 |
| 工单状态 | `业务状态 / Business Status` | 待排产、已下达、执行中、已完成、异常等状态是否统一 |
| 当前工序 | `当前工序 / Current Operation` | 发料、报工、完工是否受当前工序约束 |
| 物料准备 | `物料齐套 / Material Readiness` | 物料是否齐备 |
| 齐套状态 | `齐套状态 / Kitting Status` | 是否允许投产或批量发料 |
| 发料动作 | `物料发放 / Material Issue` | 库存不足或状态不可用时是否禁止发放 |
| 发料参数 | `发放数量`、`发放去向`、`来源库存`、`接收责任人` | 发放结果是否可追责 |

- 工单在 `异常` 或 `已完成` 状态下是否允许继续发料。
- 物料未齐套时是否允许投产。
- 发料后是否必须减少可用库存。
- 接收责任人是否必须记录。

**可落地治理规则**

| 规则类型 | 建议规则 |

| 发料前置 | 库存、发放数量、发放去向、接收责任人都确认后才能发料 |
| 库存规则 | 物料发放结果必须减少可用库存 |
| 状态规则 | 工单处于 `异常/已完成` 时，不允许继续普通发料 |
| 齐套规则 | `material_readiness = 未齐套` 时，不允许进入投产或批量发料 |

### 3.1 审计项目责任网络

**典型现象**

审计项目不是只有项目编号和项目名称，还包含项目成员、主项目经理、被审计对象、业务领域、风险评估和来源责任人。它可以用于识别组织责任是否完整。

| 治理对象 | 具体字段/关系 | 当前问题 |
| --- | --- | --- |

| 项目来源 | `项目来源 / Project Source` | 年度规划、动态立项或其他来源 |

| 项目成员 | 一个审计项目可包含多个项目成员 | 成员是否完整 |

- 项目能否启动，取决于项目经理、项目成员、被审计对象是否齐全。
- 没有主项目经理的审计项目，属于组织责任缺口。

### 3.3 技术评审角色职责边界

**典型现象**

技术评审中同一个“评审角色”可能是执行人、申请人、审批人、会签人或通知对象。如果不治理，审批链和职责边界会混乱。

| 治理对象 | 具体字段/角色 | 当前问题 |
| --- | --- | --- |
| 角色身份 | `角色名称 + 所属组织` | 是否足以唯一识别评审角色 |
| 名称  | `名称 / Name` | 角色展示和引用 |
| 业务负责人 | `业务负责人 / Business Owner` | 谁维护角色口径和责任 |
| 适用业务域 | `适用业务域 / Business Domain` | 角色属于哪个业务域 |
| 业务状态 | `业务状态 / Business Status` | 角色是否启用、停用或处于某生命周期状态 |
| 职责范围 | `responsibility scope` | 角色可以做什么 |
| 参与方式 | `executor`、`applicant`、`approver`、`countersigner`、`notification recipient` | 是执行、申请、审批、会签，还是仅通知 |

**组织优化问题**

- 同名角色在不同组织下是否是不同角色。
- 角色是否有业务负责人。
- 一个角色是否同时是申请人和审批人，是否存在职责冲突。
- 哪些角色只有通知权，哪些角色有审批或会签责任。

**可落地治理规则**

| 规则类型 | 建议规则 |
| --- | --- |
| 标准字段 | `review_role_name`、`owning_org`、`business_owner`、`business_domain`、`role_status` |
| 唯一识别 | 默认使用 `角色名称 + 所属组织` 识别评审角色 |
| 权限矩阵 | 每个角色必须标明 `executor/applicant/approver/countersigner/notified_party` 中的参与方式 |
| 责任规则 | 没有 `business_owner` 的评审角色进入责任缺口清单 |
| 冲突规则 | 同一流程中同一角色既是申请人又是审批人时，进入职责冲突检查 |
| 生命周期 | 停用角色不得继续出现在新评审流程中 |
